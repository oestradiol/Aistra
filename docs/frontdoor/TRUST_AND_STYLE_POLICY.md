# Trust and style policy

This package should read as careful work, not as inflated work.

Rules for current front-door writing:
- say plainly what the project is
- say plainly what it does not yet prove
- point to specific files instead of vague gestures
- prefer clear explanation over defensive jargon
- do not pad with visionary language, hype, or fake certainty
- do not use the amount of structure as a substitute for evidence
- keep the current docs readable by a normal careful person, not just by a compliance mindset

The package should feel:
- specific
- bounded
- edited
- accountable
- revisable

It should not feel:
- generic
- bloated
- self-congratulatory
- mechanically over-produced
- more certain than the evidence really allows


Style pressure rule:
- prefer short concrete sentences over stacked compliance wording
- prefer one clear claim over a pile of cautious synonyms
- if a sentence sounds machine-heavy, rewrite it before release
