# Anti-Slop Trust and Style Policy v0.1

This package should not ask readers to trust it because it looks polished.
It should ask readers to trust it only where the package is specific, bounded, checkable, and accountable.

## Front-door writing rules

Front-door docs should prefer:

- plain claims over inflated framing
- concrete scope over visionary language
- short direct sentences over padded abstraction
- named files and checks over vague assurances
- explicit limits over implicit certainty
- accountable authorship and provenance over anonymous polish

## Avoid these failure signals

Avoid front-door language that reads like generic synthetic filler:

- repeated claims with no new information
- hype words without operational meaning
- claims of rigor without showing the check surface
- abstract quality adjectives standing in for evidence
- long polished paragraphs that do not help the operator act
- pretending unresolved questions are already solved

## Package trust rule

A document should justify its existence by doing at least one of these jobs:

- define
- constrain
- verify
- explain current status
- tell the operator what to do next
- preserve traceability

If a file does none of those, it should be removed, merged, or archived.

## Operator-facing accountability rule

The front door should always make five things quickly visible:

1. what the project is
2. what it is useful for
3. what it does not claim
4. what the current state is
5. what to run before use
